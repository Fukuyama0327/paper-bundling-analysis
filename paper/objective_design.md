# Objective Function Design

このファイルは、査読論文・博士論文に向けて、地域分割最適化の目的関数をどのように置くかを検討するための設計メモである。

現時点では結果を確定させるのではなく、目的関数の置き方、他の候補を採らない理由、今後差し替える場合の判断基準を文章として残す。

## 1. 今回の論文の中心命題

本論文の中心は、広域連携による橋梁維持管理の発注効率化を、契約バンドリング効果として定量化することである。

ここで評価したいのは、単に橋梁を空間的に近い単位へ分けることではない。確率的に発生する補修需要を、同一地域内でどの程度同時発注として束ねられるかである。

したがって、目的関数の中心には、地域ごとの期待契約件数を置くのが自然である。

```math
\min \sum_{k \in K} f(N_k,L)
```

ここで、\(N_k\) は地域 \(k\) に含まれる橋梁数、\(L\) は1契約で同時発注できる橋梁数の上限、\(f(N_k,L)\) は閉形式で表される期待契約件数である。

## 2. 暫定主案

暫定的な主案は、距離制約付きの期待契約件数最小化である。

```math
\min \sum_{k \in K} f(N_k,L)
```

subject to:

```math
d_{ij} \le D
\quad
\text{if bridges } i \text{ and } j \text{ are assigned to the same region}
```

この置き方では、発注効率は目的関数として扱い、広域化に伴う移動距離・現場管理上の負担は制約として扱う。

この設計の利点:

- 契約バンドリングによる発注効率化という本論文の中心命題と直接対応する。
- 距離制約 \(D\) を変化させることで、広域連携範囲と発注効率のトレードオフを比較できる。
- 目的関数の単位が期待契約件数に統一され、結果の解釈がしやすい。
- 距離を目的関数に混ぜないため、契約件数と距離の重み付けを外生的に設定する必要がない。

## 3. なぜ距離を目的関数に入れないのか

候補として、距離ペナルティを目的関数に加える案がある。

```math
\min
\sum_{k \in K} f(N_k,L)
+
\lambda \cdot \text{DistancePenalty}
```

この案は、契約件数と移動距離のトレードオフを1つの目的関数で扱える利点がある。

一方で、今回の論文では主案にしない理由がある。

第一に、\(\lambda\) の設定根拠を説明する必要がある。契約件数1件の削減と、地域内距離1kmの増加をどのように換算するかは、実務上も政策上も明確ではない。重み付けを任意に置くと、分析結果が \(\lambda\) に依存し、契約バンドリング効果そのものの解釈が弱くなる。

第二に、距離を目的関数に入れると、本論文の中心である「契約バンドリングによる発注効率化」が、距離最小化や空間的まとまりの議論に埋もれやすい。

第三に、距離制約として扱えば、\(D\) を政策変数として比較できる。これは、どの程度の広域連携範囲を許容すると発注効率がどれだけ変わるかを示すうえで分かりやすい。

したがって、今回の論文では距離は目的関数ではなく制約として扱う方がよい。

## 4. なぜ地域数 \(M\) を目的関数に入れないのか

地域数にペナルティを置く案もある。

```math
\min
\sum_{k \in K} f(N_k,L)
+
\gamma M
```

または、地域数 \(M\) そのものを意思決定変数にする案もある。

しかし、今回の論文では、地域数 \(M\) は外生的に与え、複数の \(M\) を比較する方針が扱いやすい。

理由は以下である。

- 地域数は、自治体間連携の制度設計や協議単位に関わる政策的判断であり、単純な数理目的関数だけで決めにくい。
- \(\gamma\) の設定根拠を説明する必要があり、距離ペナルティと同様に任意性が生じる。
- 複数の \(M\) を比較することで、地域数を減らすことによる契約集約効果と、地域が大きくなりすぎることの関係を読み取りやすい。

したがって、地域数は目的関数に入れず、シナリオ変数として扱う。

## 5. なぜ供給制約を入れないのか

中間審査では、施工者側の供給制約、人員不足、発注効率の議論が追加された。

しかし、今回論文化するのは博士研究全体のうち第1研究であり、中心は広域連携と契約バンドリング効果の定量分析である。

供給制約を明示的に入れると、モデルは以下のように別の研究問題へ広がる。

- 地域ごとの施工者数・技術者数の推定
- 発注量と受注能力の対応
- 供給不足による未処理需要や遅延のモデル化
- 発注者側効率と受注者側能力の同時最適化

これらは重要だが、今回の論文の主張を複雑にしすぎる。

したがって、供給制約や人員不足は背景として必要最小限に触れるに留め、目的関数や制約条件には入れない。

## 6. なぜ現行管理からの改善率を直接目的関数にしないのか

現行管理との比較は、論文の結果解釈として重要である。

一方で、改善率を直接目的関数にする必要はない。

```math
\max
\frac{Z_{\text{current}} - Z}{Z_{\text{current}}}
```

ここで \(Z_{\text{current}}\) は現行管理の期待契約件数である。これは \(Z\) を最小化することと単調に対応するため、目的関数としては冗長である。

改善率は、最適化後の評価指標として示す方が分かりやすい。

## 7. なぜ閉形式 \(f(N,L)\) を使うのか

Word時点では、期待契約件数関数はシミュレーション結果に対する分数関数フィッティングとして表されていた。

しかし、査読論文化では閉形式 \(f(N,L)\) を主にする方がよい。

理由は以下である。

- \(q\) と \(L\) の意味が明示され、モデルの仮定を説明しやすい。
- 同時発注上限 \(L\) の感度分析へ拡張しやすい。
- フィッティング式の係数に依存せず、劣化確率と契約バンドリングルールから期待契約件数を導ける。
- シミュレーションとの比較により、閉形式の妥当性を確認できる。

このため、シミュレーションフィットは研究過程または比較対象として扱い、目的関数には閉形式を用いる。

## 8. 実装上の扱い

閉形式 \(f(N,L)\) は、地域内橋梁数 \(N_k\) に対する非線形関数である。

ただし、\(N_k\) は整数であり、対象橋梁数は322である。したがって、\(N=1,\ldots,322\) の全整数点で \(f(N,L)\) を事前計算し、GurobiのPWL制約として与えれば、整数点上では閉形式目的関数をそのまま表現できる。

代表ケースの検証では、20点PWLと全整数PWLで \(D=35\) km, \(M=3\) の解が異なった。そのため、今後の主結果では全整数PWLを用いる。

この知見は、目的関数の形を今後変更する場合にも重要である。目的関数を差し替える場合も、整数点での値を事前計算し、PWLまたは同等の離散表現として与える方針を基本にする。

## 9. 今後目的関数を変える場合の判断基準

目的関数を変更する場合は、以下を満たすか確認する。

1. 本論文の中心命題である「契約バンドリングによる発注効率化」と直接関係するか。
2. 追加する項の重みや単位換算に説明可能な根拠があるか。
3. 結果の解釈が、契約件数削減から離れすぎないか。
4. 供給制約や人員不足など、別研究テーマへ論点が広がりすぎないか。
5. 既存の閉形式目的関数との比較が可能か。

これらを満たさない場合は、目的関数に入れるのではなく、制約、感度分析、または考察上の限界として扱う。

## 10. 現時点の位置付け

現時点では、結果を確定させない。

暫定的には、以下を主モデル候補とする。

```math
\min \sum_{k \in K} f(N_k,L)
```

subject to:

```math
d_{ij} \le D
\quad
\text{for all bridge pairs assigned to the same region}
```

ただし、論文執筆の過程で、距離、地域数、契約上限 \(L\)、またはその他の実務的要因の扱いを再検討する余地を残す。

このメモは、目的関数を固定するためではなく、目的関数を変更する場合にも議論が散らばらないようにするための基準として使う。
